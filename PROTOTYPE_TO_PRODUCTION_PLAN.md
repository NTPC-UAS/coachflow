# CoachFlow 原型到正式版實作清單

## 專案現況

目前系統已具備可用的原型基礎，包含：

- `admin` 人員管理者控制台
- `coach` 教練工作台
- `student` 學生填寫入口
- 多教練與學生指派邏輯
- 歷史課表、歷史紀錄、批次匯入、建議重量等核心原型功能

目前版本已可作為穩定原型，但仍以本地資料與前端邏輯為主，尚未完成正式上雲與權限控管。

---

## 第一階段：資料正式上雲

這是最優先的正式化工作。

### 1. 建立正式 Google Sheets 結構

建議工作表：

- `Coaches`
- `Students`
- `Programs`
- `ProgramItems`
- `WorkoutLogs`

### 2. 將前端 `localStorage` 改為雲端資料來源

三個入口都需要改成讀寫 Google Sheets：

- `admin`
- `coach`
- `student`

### 3. 建立 Google Apps Script API

至少需要以下 API：

- 讀取教練
- 新增 / 修改 / 刪除教練
- 讀取學生
- 新增 / 修改 / 刪除學生
- 指派學生給教練
- 讀取課表
- 儲存課表
- 發布課表
- 學生送出紀錄
- 匯入歷史資料

### 4. 將登入身份改為依雲端資料判斷

不要再只依賴前端 state，而是由後端資料決定：

- 哪位教練登入
- 哪位學生登入
- 哪份課表屬於哪位教練

---

## 第二階段：角色入口正式化

### 1. `admin` 正式獨立

只做人員管理：

- 管理教練
- 管理學生
- 指派學生給教練
- 檢視最後使用狀態

### 2. `coach` 正式登入

登入方式：

- 教練代碼
- 教練專屬連結 / token

登入後只看自己的：

- 學生
- 課表
- 今日紀錄
- 歷史查詢

### 3. `student` 正式登入

登入方式：

- 學生代碼
- 學生專屬連結 / token

登入後：

- 自動讀取自己教練的目前課表
- 只看自己的歷史紀錄

---

## 第三階段：核心資料流穩定化

### 1. 課表發布機制

- 每位教練有自己的目前課表
- 學生只讀自己教練的目前課表

### 2. 紀錄覆蓋機制

對於：

- 同一位學生
- 同一份課表
- 同一個動作

若重送紀錄，應覆蓋舊資料，不重複新增。

### 3. 歷史資料匯入正式寫入 `WorkoutLogs`

功能包含：

- CSV 匯入
- 欄位驗證
- 預覽
- 匯入成功 / 失敗摘要

### 4. 建議重量正式化

規則：

- 有 `1RM` 紀錄時直接使用
- 沒有 `1RM` 時，從最近可用的 `3RM / 3x5 / 5x5` 反推
- 只顯示於 `主項目`
- 重量四捨五入到最接近的 `2.5kg`

---

## 第四階段：提升管理效率

### `admin`

- 教練停用 / 刪除後的資料處理更完整
- 未指派教練學生的篩選
- 教練 / 學生最後使用資訊更清楚

### `coach`

- 今日紀錄依日期切換
- 歷史查詢日期區間
- 匯出 CSV
- 課表模板

### `student`

- 完成後下載圖片
- 歷史紀錄查看更多
- 手機版更接近正式 app 體驗

---

## 第五階段：正式互動 UI

### 1. 將 `window.alert` / `prompt` 改為正式 UI

改成：

- toast
- modal
- 內嵌提醒

### 2. 統一錯誤與成功提示

包含：

- 必填錯誤
- 成功送出
- 匯入成功 / 失敗
- 指派成功
- 刪除確認

---

## 第六階段：完整權限控管

如果要正式長期上線，再往下做：

- 管理者真正登入
- 教練真正登入
- 學生真正登入
- token 重發 / 失效 / 重置
- 後端層級的資料權限驗證

---

## 建議實作順序

建議依序進行：

1. Google Sheets 欄位定稿
2. Apps Script API
3. `student` 接上雲端
4. `coach` 接上雲端
5. `admin` 接上雲端
6. 歷史資料匯入正式化
7. 建議重量正式化
8. 將 alert / prompt 改為正式 UI
9. 最後再做完整權限控管

---

## 一句話結論

目前這版已經是可用的穩定原型，下一步不是再堆功能，而是：

**把資料流正式後端化。**
